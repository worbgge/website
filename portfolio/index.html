Creating a website

Server Side

I've created many websites (this one included) where I have a deep knowledge of not just HTML, CSS, and JS but also the networking that also goes behind creating a website.

Video link

In this demo I show the networking side of creating a website.

First I installed nginx, which is a web server, which is the software that will be used to run my website.

Second, I start to configure nginx so that it sees where my files are being held and what ports to listen on (here, I put 80, which is the port for HTTP - later I will go through how to add HTTPS or port 443).

After configuring where to find the files for my website, I then create the website. For this tutorial I created a very simple .html file with no CSS (I extended on this further later to create this page).

After creating the HTML file and in turn creating the website, I used the command `ln -s`, which is a symbolic link that will link the two files as if they were the same file. I linked the file from sites-available to sites-enabled, which now means that the website can now be seen on our web browser.

But before opening the server, I must open the `80` and `443` ports, as these ports will be used to access the website. With those open, I can type localhost:80 to see the website.

Lastly, now that we have an HTTP server, we can enable HTTPS for the web server.

First we install certbot. I installed certbot-nginx and ran the command certbot --nginx.

Now that certbot has run, it will run out once every year. Additionally, we can create a cron job that will run every year to update this certbot.

0 0 1 1 * certbot --nginx renew

HTML and CSS

This section is a lot less server focused and a lot more focused on making a website look good.

Video link

We start the website with DOCTYPE; this just tells the browser what it should be expecting (in this case, HTML). After the doctype, you have the HTML header, and in this header we give the attribute "xmlns" which makes it conform with XHTML; we also sent the language here.

Next is the head tag, which does lots of different things:

  • Title: Gives the title for the page, which will show in the browser tab.
  • Charset: Specifies the character encoding for HTML.
  • Stylesheet: Rel is required to specify the relationship with this document and another document (in this case, style.css).
  • Viewport: This is an accessibility change that makes mobile use of the website look better.
  • Author: This isn't required, but is useful to get the contact information of the author/
  • Icon: Gives the icon, which will be used in the tab.

After the head tag (and has been closed off with /head), we get to the body tag, which shows everything that will be within the document.

We start with the pre tag, which is similar to p but for preformatted text. Here I copy in some output I get from a terminal command "neofetch" with a modified version more fit for this website.

After closing off the pre, we go on to the last two parts of the HTML document, that being both p tags to show my email address (something which is important to put on a website) and some small text that shows the last time the page was edited.

Now after finishing the HTML document, I go back and go to the style.css, which was defined earlier.

In the CSS file we first define what HTML tag we are editing, so first is body, which we give the background of #282828 and the colour of #f6e187; background defines the background colour of the element and color defines the text colour of the element.

Now that body has been set we go with two smaller ones which are `a` and `.small`; `a` is a built-in html element so we can just give that a colour meanwhile .small isn't, the `.` indicates that this is a class instead of a built-in element and we give this the `font-size:small` argument to give every tag that uses the .small class be given the small font.

The colours that were used go in line with the gruvbox colour scheme, which I think personally is the best theme that has multiple iterations of it. Now that the HTML and CSS have been written, we can see how the website looks.

WIFI Hacking

During my time at university (2023-2024), I created a Wi-Fi hacking tutorial using a MITM (Man-in-the-Middle) attack. Using the software fluxion, I described what you had to do to set up a man-in-the-middle attack.

How fluxion worked was like this: it created a copy of the network you wanted to connect to (so, for example, BTHUB6-53CZ), and when you logged into the copied version, it asked for your password and then gave you internet however, because of how the internet worked, I took the password for the internet and can now log in to the non-spoofed address.

Now that a person has connected to the network, it will save the information about that connection. At university I got a 84 out of 100 marks for this (and some other stuff).

All networking hacking was done on my own network without any illegal activity taking place.
